**Step 1.6 (population balance): looks healthy.** 28.7% face-on, 32.7% edge-on, the rest in between — nowhere near the "95% face-on, statistically dead" danger case Kimi flagged. Fine to proceed on this front.

**Step 1.1 (sign check): the result itself is plausible, but the *confidence* in it looks weak, and I think I see why.**

Physically, our derivation (and the literature statement in the original doc) predicted `combo_A` (φc+2ψ) should be the well-constrained one at face-on. The script found the opposite — `combo_B` is well-constrained. That's not automatically wrong: this is *exactly* why Step 1.1 exists (Sec. 4 of the original doc explicitly flagged that the real sign/combination depends on IMRPhenomXPHM's specific convention, which we never assumed we knew in advance). A flipped label relative to the toy-model prediction is a legitimate possible outcome.

But look at the two numbers together: a **1.2x correlation ratio** (barely distinguishable) alongside a **confirmed sign flip with `cos ι`**. That combination is a red flag for a specific, fixable issue: if which combo is "good" genuinely flips sign depending on whether `cos ι` is positive or negative — which the script itself just confirmed — then computing **one aggregate correlation across the whole population** (mixing both sign regimes together) will partially cancel the signal from each regime against the other, and you'd see exactly this kind of diluted, near-1:1 ratio even if the underlying effect is strong and clean within each half. Step 1.1 as originally specified says "conditioned on sign(cos ι)" for exactly this reason — worth checking whether the script actually split the correlation by `sign(cos ι)` before picking a single winner, or computed one number across the mixed population.

**Recommendation:** rerun with the correlation computed **separately for `cos ι > 0` and `cos ι < 0`** subsets, and report two ratios instead of one. If each subset shows a much stronger, cleaner ratio (even if `combo_A` wins in one half and `combo_B` in the other), that's a good, physically sensible result — you'd just need to track the sign-dependent label rather than a single fixed one. If splitting *still* gives weak ~1.2x ratios in both halves, that's a real concern about the harness itself, not just an aggregation artifact.

Something is inconsistent between how the endpoint summary values were computed and the polynomial coefficients that got printed alongside them — possibly the endpoints are read off the raw measured data directly while the polynomial is a separately (and badly) fit approximation, but either way this needs to be sorted out before `w(ι)` is trustworthy.

Also worth noting on principle: a raw unconstrained polynomial fit can go negative, which is meaningless for a loss weight — worth fitting something that's constrained non-negative (or fitting in a transformed space, or just using the empirical curve via interpolation instead of forcing a low-order polynomial) rather than patching sign issues after the fact.
