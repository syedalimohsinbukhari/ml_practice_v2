Good foundation, and the design decision (B) is the right call — but there are a few things I'd resolve before writing code, because a couple of them change the scope of this from "2 files" to something bigger, and one is a correctness landmine if missed.

**1. Frozen dataclass field ordering issue.** `HeadSpec` is `@dataclass(frozen=True)` with `column: int` as a required, no-default field. If you add `columns: tuple[int,...] | None = None` after it, that's fine syntactically — but `column` itself still has no default, meaning every head (including `sky_position`, which doesn't use it) must still supply *something* for `column`. You'll want to make `column: int | None = None` too, and add a `__post_init__` validation that exactly one of `column`/`columns` is set per head — otherwise you'll end up with a meaningless placeholder value for `sky_position.column` that someone could accidentally use later.

**2. Correctness risk, worth flagging loudly before implementation: does `TargetTransforms.fit()` apply generic scaling after the per-kind transform?**

`LOG_ZSCORE`/`ZSCORE` fit a mean/std from training data and rescale. If `fit()`/`transform_head()` has any shared post-processing step that z-scores *every* head's output regardless of transform kind, applying that to the `(N,3)` unit vector would independently rescale x, y, z by their own per-axis mean/std — which **destroys the unit-norm constraint** the vMF loss depends on (`μ·x` is only a valid cosine-similarity if both are actually on the unit sphere). This transform needs to be fully deterministic and bypass any generic scaling entirely — worth explicitly confirming (not assuming) that `fit()` has a no-op path for `SPHERICAL_UNIT_VECTOR`, since this would fail silently (train fine, just be geometrically wrong) rather than crash.

**3. Scope check: this is likely more than 2 files, because the head has two output tensors, not one.**

Every other head in your architecture maps to a single output tensor with a single loss. `sky_position` needs `(mu_raw, kappa_raw)` — two tensors — jointly consumed by one loss function. That has ripple effects the plan doesn't mention yet:
- `attach_heads`/`heads.py` needs to support a head producing two named output tensors, not the input-branching mechanism you already built for `q_tokens` (that solved a different problem — multiple *inputs* to one head, not multiple *outputs* from one head).
- `losses.py`'s per-head loss table presumably does `loss_fn(y_true[head], y_pred[head])` — `vmf_nll_loss` needs `(y_true, mu_raw, kappa_raw)`, a different signature shape than every other entry in that table.
- **Metrics/diagnostics are the biggest gap.** Your entire diagnostic stack — `r2_*`, `mae_*`, `std_ratio_*`, the subset callbacks, `residuals_vs_param` — assumes a scalar-ish per-head prediction comparable via R²/MAE. R² and per-axis MAE on 3 unit-vector components isn't a meaningful metric (the components are constrained, not independent), and `std_ratio` as currently defined doesn't map onto κ at all. You'll want a dedicated `angular_separation`-based metric (already sketched) as the actual headline number for this head, plus something to sanity-check κ calibration — this needs to be a genuinely new code path in `callbacks.py`/`evaluate.py`, not a value slotted into the existing generic metrics loop.

**4. Before deleting `ra`/`declination` from `HEAD_SPECS`: grep the whole repo first.** Configs, `evaluate.py`, and diagnostic filters like `sigmoid_logit_hist` (which selects heads by `UNIT_AFFINE`+sigmoid — `declination` currently matches that filter) all reference these head names directly. A hard removal will silently break anything still pointing at the old names rather than erroring clearly, unless `resolve_heads()`'s unknown-head check catches it — worth confirming it does before relying on it.

**Recommendation on sequencing:** I'd implement this in two passes rather than one big-bang integration — (1) get `heads_spec.py` + `transforms.py` + `attach_heads` + `losses.py` wired up with a **standalone eval script** (not the generic diagnostics pipeline) that just prints angular error and κ stats, confirm it trains and the numbers look sane on real data; *then* (2) fold it into the generic `callbacks.py`/`diagnostics.csv` machinery once you trust the mechanics. That avoids debugging "is the vMF math wrong" and "is the generic metrics pipeline choking on a 2-tensor head" at the same time.

Otherwise, the core idea and file-1 design are solid — go ahead on `heads_spec.py`/`transforms.py`, just build the output/metrics side as a real, separately-scoped step rather than an afterthought.
